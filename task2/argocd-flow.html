Before the flow

Meet the five components

Git Server

Hosts the desired-state manifests (plain YAML, Helm chart + values, or Kustomize). Any git host — GitHub, GitLab, etc. Not part of Argo CD itself; it's just the source of truth.

argocd-server

The API/UI/CLI gateway. Enforces RBAC and SSO, serves status to users, and receives git webhooks. It does not run the reconciliation loop itself.

application-controller

The actual reconciliation engine. Watches Application custom resources, diffs desired vs. live state, and applies changes to the target cluster.

repo-server

Stateless manifest renderer. Clones/fetches from the git server and runs the right templating tool (Helm/Kustomize/plain) to produce final Kubernetes YAML. Caches results.

Kubernetes API Server

The shared source of truth every other component talks to — it stores both the live cluster resources being managed and Argo CD's own Application custom resources (as etcd-backed CRs).

Walk through it, step by step

Step 1

A change lands in Git

A developer merges updated manifests (or Helm values) into the tracked path. That commit is now the desired state Argo CD must reconcile the cluster toward.

Step 2

A refresh is triggered

Either a git webhook hits argocd-server's webhook endpoint (near-instant), or application-controller's own timer polls on its default ~3-minute interval. Either path ends the same way: the Application resource is marked for refresh.

Step 3

application-controller picks it up

That "refresh" is really just a write to the Application custom resource, stored via the kube API server. application-controller watches Applications through a kube API informer, so it notices the refresh instantly — regardless of who requested it.

Step 4

repo-server renders the manifests

application-controller calls repo-server's GenerateManifest gRPC API. repo-server fetches the target commit from the git server and runs the right tool (Helm/Kustomize/plain YAML) to produce final Kubernetes manifests.

Step 5

Controller diffs desired vs. live

application-controller already keeps a cache of the target cluster's live objects, built from kube API server watches. It normalizes and diffs that live state against repo-server's rendered manifests to compute exactly which resources are OutOfSync.

Step 6

Controller applies the diff

Using its own ServiceAccount credentials, application-controller applies creates/updates/prunes directly against the kube API server (server-side apply, if enabled). Native Kubernetes controllers then roll out the actual change.

Step 7

Status is reported, loop continues

application-controller writes the new Sync/Health status back onto the Application resource. argocd-server just reads that status for the UI/CLI. The loop never stops — the next drift, from git or a manual kubectl edit, gets caught the same way.

Why this matters

It never really stops.

At the heart of GitOps.
  • Default polling: the application-controller re-diffs every Application on a timer (~3 minutes by default).
  • Webhooks are an alternative to polling: they just skip the polling wait and funnel into the exact same diff/apply logic. They help avoid rate limiting by the Git provider.
  • Self-heal: with selfHeal: true, a manual kubectl edit that drifts from git gets reverted on the very next loop — live state no longer matches the rendered manifest, so the controller "fixes" it back to git. One of the tenets of GitOps.
  • Two independent status fields: Sync status (does live match git?) and Health status (are the resulting objects actually healthy?) are separate. A Synced app can still be Degraded (e.g. CrashLoopBackOff) — sync says nothing about whether the change worked.

Typical points where things could break.

Git Server Repo unreachable or auth failure Application stuck in an error/Unknown state; check the repo credentials Secret and repo-server logs (clone/fetch errors).
argocd-server Webhook never arrives App only updates on the ~3-min poll instead of instantly; check the webhook secret/URL config and argocd-server webhook logs.
repo-server Render failure (bad chart, wrong Helm/Kustomize version, missing values) Application shows a ComparisonError; check repo-server logs and argocd app manifests.
application-controller Stuck OutOfSync or diffing forever Often a CRD-before-CR ordering problem; check sync-wave annotations and controller logs.
kube API server Apply fails with RBAC/admission errors Controller logs show the raw Kubernetes error; check the Argo CD ServiceAccount's RBAC and any admission webhooks in the target cluster.

Key takeaways

Four things worth remembering

application-controller is the reconciliation engine. argocd-server is API/UI/webhook plumbing, not the loop itself.
The kube API server is the shared backbone: it stores both Argo CD's own state (Applications) and the managed cluster's live state.
Auto-sync is a continuous loop, not a one-time action — every poll, webhook, and self-heal pass runs the same diff/apply path.
The repo-server does the manifests generation, using either helm or kustomize (mainly). It's also where we can inject extras like decryption, secrets management and more, via the Config Management Plugins (CMP).
